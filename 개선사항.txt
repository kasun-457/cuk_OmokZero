5시간 20분 결과를 자세히 봤는데, **심각한 문제가 있어요. 지금 이 학습은 사실상 전혀 발전하지 않고 있어요.** 솔직하게 말씀드릴게요.

## 무슨 일이 벌어지고 있나

두 가지 결정적 증상이 보여요:

1. **`pi_loss`가 4.83 → 4.80으로 거의 안 움직여요.** `log(121) = 4.795`인데, 이건 정책망이 **완전 균일분포(=학습 안 됨)** 수준에 머물러 있다는 뜻이에요.
2. **매 iteration "Previous model kept"** → 승률이 한 번도 0.55를 못 넘어서 `best_model.pt`가 **아직 한 번도 저장 안 됐어요.**

원인은 `train.py`의 학습 루프 구조예요:

```python
self.net.train(training_data)      # 학습해서 가중치 개선
...
if win_rate > 0.55:
    self.net.save_model("best_model")
else:
    self.net.load_model()          # ← 학습 결과를 통째로 되돌림!!
```

승률이 0.55를 못 넘으면 **그 iteration에서 학습한 걸 전부 버리고 직전 가중치로 되돌려요.** 그런데 14게임짜리 노이즈 많은 평가에서 갓 학습한 모델이 0.55를 안정적으로 넘기는 건 거의 불가능해요. 결과적으로:

- 매번 학습 → 평가 실패 → 되돌림 → 다음 iteration도 거의 같은 출발점
- 정책망이 균일분포에서 못 벗어남
- 5시간 20분 동안 누적된 발전이 0에 가까움

이건 제가 처음에 config만 보고 놓친 부분이에요. 원래 AlphaGo Zero 논문의 게이팅 방식인데, 작은 평가셋 + 짧은 시간 예산에선 학습을 죽여버려요. 현대 AlphaZero는 이 게이트를 빼고 **항상 누적**하는 방식을 써요.

이걸 고칠게요. 현재 train.py 루프를 다시 확인하고 수정합니다.

이제 한 가지 더 — self-play가 iteration당 ~42분으로 너무 느려요(초반 정책이 무작위라 게임이 거의 121수까지 끌려서). 이제 학습이 실제로 누적되니 **iteration 수를 더 확보**하는 게 중요해요. sims를 낮춰 처리량을 올리고, 누적 학습이 더 잘 되도록 epoch를 늘릴게요.

이제 수정한 코드의 문법을 검증하고, 진단대로 `best_model.pt`가 실제로 없는지 확인할게요.

확인됐어요 — **`best_model.pt`가 아예 없어요.** `current_model.pt`만 있고요. 진단이 맞았어요: 5시간 20분 동안 매 iteration 학습이 평가 게이트에서 되돌려져서, 쓸 만한 모델이 하나도 안 나왔어요.

재시작 시 동작을 정확히 안내하려고 `main.py`의 모델 로딩 로직을 확인할게요.

`main.py`는 `best_model.pt`가 있을 때만 이어서 학습하는데, 그게 없으니 **재시작하면 자동으로 처음부터** 시작해요. 어차피 지금 `current_model`은 균일분포 수준이라 새로 시작해도 잃는 게 거의 없어요.

## 무엇을 고쳤나

**핵심 (train.py) — 학습 폐기 버그:**
- 평가 승률이 0.55 미만이면 학습 결과를 통째로 되돌리던 로직 제거
- 이제 **매 iteration 학습 결과를 항상 채택·누적**(`current_model`), 평가는 진척도 로깅과 `best_model` 추적용으로만 사용 → 정책망이 균일분포에서 벗어날 수 있음

**config.py — 더 많은 iteration 확보:**
| 항목 | 변경 | 이유 |
|---|---|---|
| `num_mcts_sims` | 256 → **160** | iter당 42분이 너무 느림. 학습 누적이 가능해진 만큼 iteration 수가 더 중요 |
| `epochs` | 4 → **6** | self-play가 시간 대부분이라 epoch 증가는 거의 공짜. 누적 버퍼에서 더 학습 |

| `lr_milestones` | [10,16,20] → **[6,10,13]** | 도달 가능한 ~12 iter에 맞춤 |
